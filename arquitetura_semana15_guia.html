01 · Comparação de Técnicas
✏️ Exercício 1 — Comparação de Técnicas de I/O
📋 Objetivo: Avaliar Polling × Interrupção × DMA
⏱ 30 min
CritérioInsuficiente (0–5)Básico (6–7)Proficiente (8–9)Excelente (10)
Tabela ComparativaMenos de 2 técnicas ou incompleta3 técnicas com dados superficiaisTabela completa com critérios relevantesInclui dados numéricos (% CPU, MB/s)
Uso de CPU (Polling)Não menciona busy-waitMenciona sem quantificarExplica que CPU fica 100% ocupadaCalcula ciclos desperdiçados por byte transferido
Mecanismo de InterrupçãoConfunde com pollingDescreve superficialmenteDescreve salvar contexto → ISR → restaurarMenciona overhead de interrupção e latência de resposta
Vantagens do DMANão identifica vantagemDiz que libera a CPUExplica transferência direta mem↔dispositivoAnalisa bus contention e DMA cycles stealing
Cenário de UsoSem indicação de cenáriosCenários muito genéricosCenário específico para cada técnicaJustifica com base em taxa de dados e latência exigida
💡 ORIENTAÇÕES E DICAS
  • Polling: simples, CPU fica presa em loop. Ideal para: dispositivos muito rápidos ou sistemas embarcados simples
  • Interrupção: CPU faz outras tarefas, mas há overhead de contexto (~100 ciclos). Ideal para: teclado, mouse, eventos esporádicos
  • DMA: CPU configura e esquece. Ideal para: transferências grandes (disco, rede, áudio). Custo: controlador extra
  • Para 1MB a 100 MB/s com polling: CPU ocupada por 10ms — isso é ~20 milhões de ciclos a 2GHz desperdiçados!

Tratamento de IRQ TecladoNão representa o fluxoRepresenta parcialmenteSave context → ISR → restore → returnInclui latência de reconhecimento do controlador
Preempção pelo DiscoNão trata prioridadeMenciona sem representarMostra ISR do teclado interrompida pelo discoDiscute níveis de interrupção (PIC/APIC) e mascaramento

💡 ORIENTAÇÕES E DICAS
  • Use uma linha do tempo horizontal: t=0ms (programa), t=5ms (IRQ teclado), t=7ms (IRQ disco)
  • Salvar contexto = empilhar registradores + PC na stack
  • Disco tem prioridade maior → pode interromper a ISR do teclado (interrupção aninhada)
  • Ordem de retorno: ISR disco finaliza → continua ISR teclado → continua programa original
  • A instrução IRET/RTI restaura o contexto e retorna ao ponto exato de interrupção

✏️ Exercício 3 — Projeto de Interface I2C
📋 Objetivo: Especificar controlador de sensor de temperatura
⏱ 25 min
CritérioInsuficiente (0–5)Básico (6–7)Proficiente (8–9)Excelente (10)
Endereço I2CNão especificaEndereço sem justificativaEndereço de 7 bits com formato corretoExplica bit R/W e processo de arbitração
Mapa de RegistradoresSem registradores1 ou 2 registradores apenasStatus, Data e Config com bits descritosCada bit de cada registrador descrito e justificado
Protocolo de LeituraSem protocoloEtapas incompletasSTART → ADDR+W → REG → ADDR+R → DATA → STOPInclui ACK/NACK e tratamento de erro
Interface com Driver SOSem considerar o SOMenciona ioctl superficialmenteDescreve chamadas do driver: open, read, ioctlProjeta struct de dados e interface sysfs/devfs
Criatividade TécnicaCópia sem adaptaçãoEspecificação básica funcionalInclui modo de alerta por thresholdAdiciona modo de economia de energia e taxa configurável
💡 ORIENTAÇÕES E DICAS
  • I2C usa 2 fios: SDA (dados) e SCL (clock). Endereços são de 7 bits (0x00–0x7F)
  • Exemplo real: sensor LM75 usa endereço 0x48, registrador 0x00 = temperatura em 0.5°C steps
  • Protocolo de leitura: START, envie addr+W, envie número do registrador, repeated START, envie addr+R, leia byte, STOP
  • Registradores sugeridos: 0x00=Temp (16 bits, MSB first), 0x01=Config (bit 0 = shutdown mode), 0x02=T_high, 0x03=T_low
  • O driver do SO lê via i2c_smbus_read_word_data() no Linux
