01 · Bluetooth Serial
✏️ Exercício 1 — Comunicação Serial via Bluetooth
📋 Objetivo: Estabelecer canal bidirecional ESP32 ↔ smartphone
⏱ 30 min
CritérioInsuficiente (0–5)Básico (6–7)Proficiente (8–9)Excelente (10)
Pareamento BTNão aparece na lista de dispositivosAparece mas falha no pareamentoPareia com sucesso e exibe "Cliente conectado!" no serialTrata evento de desconexão e aguarda nova conexão sem reiniciar
Recepção de ComandosNão recebe dadosRecebe mas não processaInterpreta LED_ON, LED_OFF e STATUS corretamenteParser robusto: case-insensitive, ignora espaços e quebras de linha
Resposta ao CelularSem resposta enviadaEnvia resposta mas sem conteúdo útilResponde com estado atual após cada comandoInclui timestamp (uptime), RSSI e temperatura interna na resposta
Controle do LEDLED não respondeLED responde mas sem confirmação BTLED liga/desliga e ESP32 confirma via BluetoothFeedback de blink diferente para cada comando (1 pisca=ON, 2 piscas=OFF)
Relação com TeoriaSem conexão teóricaMenciona "comunicação sem fio"Compara SPP Bluetooth com UART (mesma abstração, meio diferente)Explica como o controlador BT usa DMA para não bloquear a CPU Xtensa
💡 ORIENTAÇÕES E DICAS
  • Biblioteca: #include "BluetoothSerial.h" — não é compatível com BLE, é Bluetooth Classic
  • Iniciar: BluetoothSerial SerialBT; SerialBT.begin("ESP32-ARQ");
  • Receber: if (SerialBT.available()) { String cmd = SerialBT.readStringUntil('\n'); cmd.trim(); }
  • Enviar resposta: SerialBT.println("LED: ON | Uptime: " + String(millis()/1000) + "s");
  • App Android gratuito: "Serial Bluetooth Terminal" (Kai Morich). Selecione o dispositivo "ESP32-ARQ" após pareamento no Bluetooth do celular
#include "BluetoothSerial.h"
BluetoothSerial SerialBT;
SerialBT.begin("ESP32-ARQ");
if (SerialBT.available()) {
  String cmd = SerialBT.readStringUntil('\n');
  cmd.trim();
  if (cmd == "LED_ON") { digitalWrite(2, HIGH); SerialBT.println("OK: LED ON"); }
  else if (cmd == "LED_OFF") { digitalWrite(2, LOW); SerialBT.println("OK: LED OFF"); }
  else if (cmd == "STATUS") { SerialBT.println("UP:" + String(millis()/1000) + "s"); }
}
02 · Controle Remoto BT
✏️ Exercício 2 — Controle Remoto com Watchdog
📋 Objetivo: Controle PWM bidirecional com segurança por timeout
⏱ 35 min
CritérioInsuficiente (0–5)Básico (6–7)Proficiente (8–9)Excelente (10)
Controle PWMSem controle de intensidadePWM fixo, não responde a comandosComando numérico (0–255) controla ledcWrite() corretamenteRampa suave entre valores para evitar saltos bruscos
Leitura ADC → BTNão envia ADCEnvia valor bruto sem conversãoEnvia tensão convertida (0–3.3V) ao celular a cada 2sEnvia também média de 5 amostras e desvio padrão
Watchdog por TimeoutSem watchdogTem timer mas não desliga nadaApós 10s sem comando, desliga LED e exibe aviso no serialWatchdog reiniciável via comando RESET; log de ativações no serial
RobustezTrava com comandos inválidosIgnora comandos inválidos silenciosamenteResponde "ERRO: comando desconhecido" para entradas inválidasValida range (0–255) e retorna "ERRO: valor fora do range 0-255"
DocumentaçãoSem documentaçãoCódigo sem comentáriosCódigo comentado com protocolo descrito no cabeçalhoInclui diagrama de estados do sistema (conectado/desconectado/watchdog)
💡 ORIENTAÇÕES E DICAS
  • PWM: ledcSetup(0, 5000, 8); ledcAttachPin(2, 0); depois ledcWrite(0, valor); onde valor = 0–255
  • Watchdog com millis(): if (millis() - lastCmd > 10000) { ledcWrite(0, 0); SerialBT.println("WATCHDOG: desligado!"); }
  • Parse de número: int valor = cmd.toInt(); — retorna 0 se não for número, então valide com if (valor >= 0 && valor <= 255)
  • ADC periódico sem delay: use millis() para enviar leitura a cada 2000ms independente dos comandos
  • Conexão com teoria: o watchdog é um mecanismo de timeout por hardware (Timer periférico) — o mesmo conceito de DMA timeout!
03 · Rede de Sensores
✏️ Exercício 3 — Rede P2P com ESP-NOW
📋 Objetivo: Comunicação direta entre dois ESP32s sem infraestrutura
⏱ 25 min
CritérioInsuficiente (0–5)Básico (6–7)Proficiente (8–9)Excelente (10)
Configuração ESP-NOWNão configura ESP-NOWInicializa mas sem peer registradoSender registra MAC do receiver; comunicação funcionalTroca bidirecional: receiver envia ACK com seu ADC de volta
Estrutura de DadosEnvia string não estruturadaUsa struct mas sem campos relevantesstruct com id, valor ADC, timestamp e contador de pacotesCRC simples para detecção de corrupção de pacote
Taxa de PerdaSem medição de perdaConta pacotes mas não calcula %Calcula % de perda: (enviados - recebidos) / enviados × 100Gráfico de barras ASCII no serial mostrando evolução da perda
Conexão com TeoriaSem conexão teóricaMenciona que é "sem fio"Compara ESP-NOW (sem ACK físico) com I2C (com ACK no bit 9)Mapeia protocolo ESP-NOW na hierarquia de protocolos estudada
DocumentaçãoSem documentaçãoApenas códigoLog serial estruturado com RSSI e estatísticasRelatório comparando ESP-NOW vs MQTT vs BT em tabela com latência medida
💡 ORIENTAÇÕES E DICAS
  • MAC do receiver: execute Serial.println(WiFi.macAddress()); no receiver para descobrir o endereço
  • ESP-NOW sender: esp_now_init(); esp_now_add_peer(&peerInfo); esp_now_send(mac, (uint8_t*)&data, sizeof(data));
  • Callback de recepção: void onDataRecv(const uint8_t *mac, const uint8_t *data, int len) { memcpy(&received, data, sizeof(received)); }
  • Comparação com I2C: I2C usa ACK (bit 9 da transmissão), ESP-NOW por padrão é "fire and forget" — sem garantia de entrega!
  • Alcance típico: ~200m em espaço aberto, ~20m através de paredes — muito maior que Bluetooth Classic (10m)
