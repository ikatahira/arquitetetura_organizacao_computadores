02 · Conceito-Chave
📐 TIMER1 COMO RÉGUA DE NANOSEGUNDOS
O ATmega328P opera a 16 MHz — cada ciclo de clock dura 62,5 ns. O Timer1 é um contador de 16 bits que incrementa a cada pulso de clock (prescaler = 1). Ao registrar o valor do contador antes e depois de um trecho de código, obtemos diretamente o número de ciclos gastos.

T = ciclos × (1 / f_clock) = ciclos × 62,5 ns

Isso transforma o Arduino em um analisador de desempenho em hardware real — sem simulação, sem aproximação: ciclos reais medidos em silício real. O ATmega executa a maioria das instruções em 1–2 ciclos, mas operações como divisão de inteiros consomem até 18–20 ciclos. Essa diferença, invisível no código-fonte, torna-se concreta aqui.

04 · Referência Rápida — Timer1 do ATmega328P
⚙️ Registradores e Configuração do Timer1
RegistradorFunção
TCCR1ATimer/Counter Control Register A — modo de operação
TCCR1BTimer/Counter Control Register B — prescaler e modo
TCNT1Valor atual do contador de 16 bits (0–65535)
TIMSK1Máscara de interrupções do Timer1
CS10Clock Select bit 0 — prescaler = 1 (sem divisão)
Modo Normal (WGM = 0000):
O contador conta de 0 até 65535 e sobrescreve. Com prescaler = 1 e clock de 16 MHz, cada tick = 62,5 ns. O contador completa um ciclo completo em ≈ 4,096 ms.

Atenção ao overflow:
Se o trecho medido levar mais de 4,096 ms (65535 ciclos), o contador vai dar overflow. Para trechos mais longos, use prescaler = 8 (cada tick = 500 ns) ou verifique a flag TOV1.

Fórmula de cálculo:
ciclos = t_fim − t_inicio
tempo_ns = ciclos × 62,5
tempo_µs = ciclos / 16
05 · Roteiro do Laboratório
1
Configurar o Timer1 e criar a função de medição
O primeiro sketch configura o Timer1 em modo normal com prescaler = 1 (sem divisão de clock). A macro MEDIR(expr) registra o contador antes e depois da expressão, retornando o número exato de ciclos gastos. Copie, carregue e abra o Serial Monitor a 9600 baud.
lab06_timer1_base.ino Arduino C++ 
// ═══════════════════════════════════════════
// LAB 06 — Medindo Ciclos de Clock com Timer1
// ARQ-001 · FATEC · ADS 2026
// ═══════════════════════════════════════════

// ── Configuração do Timer1 ──────────────────
void iniciarTimer1() {
  TCCR1A = 0;          // modo normal
  TCCR1B = 0;          // zera prescaler
  TCNT1  = 0;          // reseta contador
  TCCR1B |= (1 << CS10); // prescaler = 1 → clock direto 16 MHz
}

// ── Lê o contador atual ─────────────────────
inline uint16_t lerCiclos() {
  return TCNT1;
}

// ── Macro de medição ────────────────────────
// Uso: uint16_t c = MEDIR( codigo_aqui );
#define MEDIR(expr)                         \
  ([&]() -> uint16_t {                      \
    TCNT1 = 0;                               \
    uint16_t t0 = TCNT1;                    \
    (expr);                                  \
    uint16_t t1 = TCNT1;                    \
    return (t1 >= t0) ? (t1 - t0) : t1;    \
  }())

void setup() {
  Serial.begin(9600);
  iniciarTimer1();
  Serial.println("=============================");
  Serial.println("Lab 06 — Ciclos de Clock");
  Serial.println("ATmega328P @ 16 MHz");
  Serial.println("1 ciclo = 62.5 ns");
  Serial.println("=============================");
}

void loop() {
  // Teste inicial: atribuição simples
  volatile int x;
  uint16_t ciclos = MEDIR( x = 42 );

  Serial.print("Atribuicao (x=42): ");
  Serial.print(ciclos);
  Serial.println(" ciclos");

  delay(2000);
}
💡 POR QUE volatile?
A palavra-chave volatile impede que o compilador otimize e elimine a atribuição, o que zeraria artificialmente os ciclos. Em medições de desempenho, sempre declare as variáveis como volatile para garantir que o código realmente seja executado no hardware.

Experimento 1 — Custo de operações aritméticas
Meça e compare o número de ciclos de quatro operações aritméticas: soma, subtração, multiplicação e divisão. O resultado revela diretamente como o hardware do ATmega trata cada operação no nível de microcódigo.
lab06_exp1_aritmetica.ino Arduino C++ 
// (inclua a configuração do Timer1 do Passo 1)

void experimento1_aritmetica() {
  volatile int a = 100, b = 7, r;
  uint16_t c;

  Serial.println("\n--- EXP 1: Operacoes Aritmeticas ---");

  c = MEDIR( r = a + b );
  Serial.print("Soma (a+b):          ");
  Serial.print(c); Serial.println(" ciclos");

  c = MEDIR( r = a - b );
  Serial.print("Subtracao (a-b):     ");
  Serial.print(c); Serial.println(" ciclos");

  c = MEDIR( r = a * b );
  Serial.print("Multiplicacao (a*b): ");
  Serial.print(c); Serial.println(" ciclos");

  c = MEDIR( r = a / b );
  Serial.print("Divisao (a/b):       ");
  Serial.print(c); Serial.println(" ciclos");

  // Calcule o tempo em nanosegundos
  // T(ns) = ciclos * 62.5
}
📟 RESULTADO ESPERADO NO SERIAL MONITOR
--- EXP 1: Operacoes Aritmeticas ---
Soma (a+b):           2 ciclos
Subtracao (a-b):      2 ciclos
Multiplicacao (a*b): 9 ciclos
Divisao (a/b):       18 ciclos
// valores podem variar ±1-2 ciclos pelo overhead de medição
Análise: Soma e subtração são operações de 1 ciclo real no AVR (instrução ADD / SUB = 1 ciclo cada). A multiplicação usa a instrução MUL de 2 ciclos do ATmega, mas o compilador pode gerar múltiplas instruções para inteiros de 16 bits. A divisão não existe como instrução AVR — é implementada via software por uma rotina de subtração repetitiva, daí os ~18 ciclos. Isso é CPI diferenciado por tipo de instrução!

Experimento 2 — Custo de acesso à memória
Compare o custo de acessar uma variável local (registrador/stack), um array na SRAM e a EEPROM. Este experimento demonstra empiricamente a hierarquia de memória do ATmega e como latência de acesso varia com o tipo de memória.
lab06_exp2_memoria.ino Arduino C++ 
#include <EEPROM.h>

volatile int var_global = 0;        // SRAM — variável global
int array_sram[16];                   // SRAM — array na heap

void experimento2_memoria() {
  volatile int var_local = 0;        // Stack (mais próxima do CPU)
  uint16_t c;

  Serial.println("\n--- EXP 2: Acesso a Memoria ---");

  // Variável local (stack / possivelmente registrador)
  c = MEDIR( var_local++ );
  Serial.print("Var local (stack):   ");
  Serial.print(c); Serial.println(" ciclos");

  // Variável global (SRAM endereço fixo)
  c = MEDIR( var_global++ );
  Serial.print("Var global (SRAM):   ");
  Serial.print(c); Serial.println(" ciclos");

  // Acesso a array (endereço calculado)
  c = MEDIR( array_sram[8] = 42 );
  Serial.print("Array SRAM[8]:       ");
  Serial.print(c); Serial.println(" ciclos");

  // EEPROM — memória não-volátil (muito mais lenta!)
  c = MEDIR( EEPROM.write(0, 42) );
  Serial.print("EEPROM.write():      ");
  Serial.print(c); Serial.println(" ciclos (overflow!)");

  Serial.println("  → EEPROM leva ~3.3ms = 52800 ciclos");
  Serial.println("  → Timer1 deu overflow (max 65535)");
}
📟 RESULTADO ESPERADO NO SERIAL MONITOR
--- EXP 2: Acesso a Memoria ---
Var local (stack):   2 ciclos
Var global (SRAM):   3 ciclos
Array SRAM[8]:       5 ciclos
EEPROM.write():      overflow
  → EEPROM leva ~3.3ms = 52800 ciclos
// EEPROM excede o range do Timer1 de 16 bits
Análise: A hierarquia se manifesta em ciclos: stack local é mais rápida porque o compilador tende a usar registradores; variável global exige busca em endereço absoluto na SRAM; o array adiciona o cálculo do índice; a EEPROM usa um protocolo de escrita com temporização interna de 3,3 ms — equivalente a 52.800 ciclos — ilustrando de forma dramática o que é latência de memória não-volátil.

Experimento 3 — Loops e contagem de instruções
Meça o overhead de um loop for e compare o custo por iteração. Isso permite estimar o CPI médio real do programa e contrastar com o CPI teórico do ATmega (≈ 1,3 CPI para código típico).
lab06_exp3_loops.ino Arduino C++ 
void experimento3_loops() {
  volatile long soma = 0;
  uint16_t c;
  const int N = 100;

  Serial.println("\n--- EXP 3: Custo de Loops ---");

  // Loop simples: soma acumulada
  soma = 0;
  TCNT1 = 0;
  uint16_t t0 = TCNT1;
  for (int i = 0; i < N; i++) { soma += i; }
  uint16_t t1 = TCNT1;
  c = t1 - t0;

  Serial.print("Loop 100x (soma+=i): ");
  Serial.print(c); Serial.print(" ciclos totais | ");
  Serial.print(c / N); Serial.println(" ciclos/iteracao");

  // Loop com multiplicação interna
  soma = 0;
  TCNT1 = 0;
  t0 = TCNT1;
  for (int i = 0; i < N; i++) { soma += i * 3; }
  t1 = TCNT1;
  c = t1 - t0;

  Serial.print("Loop 100x (soma*=3): ");
  Serial.print(c); Serial.print(" ciclos totais | ");
  Serial.print(c / N); Serial.println(" ciclos/iteracao");

  // Calcular CPI médio estimado
  // Assumindo ~5 instruções por iteração (contador, cmp, soma, store, branch)
  Serial.print("CPI estimado (soma): ");
  Serial.println(((float)(c / N)) / 5, 2);
}
💡 DESAFIO — Otimização
Tente substituir soma += i * 3 por soma += i + i + i (3 somas em vez de 1 multiplicação). Meça os ciclos. O resultado foi mais rápido ou mais lento? Por quê? Discuta com a turma como o compilador (com -O2) lida com essa otimização de forma automática.

Análise e cálculo de CPI — Tabela de resultados
Com os dados coletados, preencha a tabela abaixo e calcule o CPI de cada operação. Compare com os valores teóricos do datasheet do ATmega328P.
📐 Fórmula de Tempo
T = N_ciclos × (1/16MHz)
T(ns) = N_ciclos × 62,5
T(µs) = N_ciclos ÷ 16

Exemplo: 18 ciclos na divisão
→ T = 18 × 62,5 = 1.125 ns
📐 Cálculo do CPI
CPI = ciclos / N_instruções

Para estimar N_instruções de um trecho, compile com avr-objdump -d e conte as instruções assembly geradas. Um compilador sem otimização gera ~3–5 instruções por linha C++.
📐 Desempenho Relativo
Speedup = T_lento / T_rápido

Calcule o speedup da soma em relação à divisão. Depois calcule quanto tempo um loop de 1 milhão de divisões levaria comparado a um loop de 1 milhão de somas.
