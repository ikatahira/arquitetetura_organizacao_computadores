01 · Planejamento do Projeto
✏️ Exercício 1 — Definição de Escopo e Requisitos
📋 Objetivo: Documentar o projeto com clareza antes de implementar
⏱ 30 min

ViabilidadeProjeto impossível no prazoProjeto grande sem priorizaçãoEscopo realista com MVP definido para a aula 20MVP + features opcionais organizados por prioridade (MoSCoW)
💡 ORIENTAÇÕES E DICAS
  • MVP (Minimum Viable Product): o que é o mínimo que precisa funcionar na apresentação? Defina isso primeiro
  • Sugestões de projetos viáveis em 2 aulas: estação climática (temp + umidade → ThingSpeak), monitor de presença (PIR + LED + BT alerta), irrigação automática (solo seco → bomba liga), ou semáforo inteligente (conta carros via sensor ultrassônico)
  • Diagrama de blocos: sensor → ESP32 → [WiFi/BT] → [nuvem/celular/outro ESP32] → [dashboard/LED/display]
  • FreeRTOS tasks ajudam a separar responsabilidades: Task1=leitura, Task2=comunicação, Task3=interface
  • Documente tudo desde o início — na apresentação, mostrar o processo é tão importante quanto o produto
02 · Implementação
✏️ Exercício 2 — Desenvolvimento e Integração
📋 Objetivo: Construir o sistema integrando subsistemas das semanas 16–18
⏱ 40 min
CritérioInsuficiente (0–5)Básico (6–7)Proficiente (8–9)Excelente (10)
Integração de MódulosApenas 1 tecnologia utilizada2 tecnologias, mas isoladas3+ tecnologias integradas e funcionando juntas4+ tecnologias com fluxo de dados coerente de ponta a ponta
Qualidade do CódigoCódigo todo em loop() sem organizaçãoFunções separadas mas sem coesãoCódigo modular com funções e comentários clarosFreeRTOS tasks com prioridades definidas e sem busy-wait
Tratamento de ErrosSem tratamento de errosApenas Serial.println() de errosReconexão automática de WiFi/BT e fallback para erroWatchdog timer de hardware + reset controlado com log de causa
Funcionamento ContínuoTrava em minutosFunciona mas com delay() em todo lugarFunciona por 10+ minutos sem travar com millis() correto30+ minutos de uptime; loop() sem delay() algum
Conexão com ConteúdoSem menção à arquitetura de computadoresMenciona ESP32 e WiFi superficialmenteRelaciona escolhas de design a conceitos de I/O e arquiteturaExplica como dual-core e FreeRTOS aplicam paralelismo estudado
💡 ORIENTAÇÕES E DICAS
  • FreeRTOS no ESP32: xTaskCreate(taskFunc, "Nome", 4096, NULL, 1, NULL); — a prioridade 1 é baixa, 5 é alta
  • Não misture Serial e SerialBT no mesmo task — use mutexes ou filas (xQueueCreate) para comunicação entre tasks
  • Reconexão WiFi automática: verifique WiFi.status() em task dedicada e chame WiFi.reconnect() se desconectado
  • Watchdog de hardware: esp_task_wdt_init(30, true); esp_task_wdt_add(NULL); + esp_task_wdt_reset(); no loop
  • Conexão com teoria: FreeRTOS é um SO de tempo real — o scheduler é o "despachante de processos" da aula de sistemas operacionais!
// Task de leitura de sensor (Core 0)
void taskSensor(void *param) {
  for(;;) {
    float temp = readTemperature();
    xQueueSend(dataQueue, &temp, 0);
    vTaskDelay(pdMS_TO_TICKS(2000));
  }
}
// Task de comunicação WiFi (Core 1)
void taskComm(void *param) {
  for(;;) {
    float val;
    if (xQueueReceive(dataQueue, &val, portMAX_DELAY)) sendToCloud(val);
  }
}

💡 ORIENTAÇÕES E DICAS
  • Modelo de caso de teste: ID | Descrição | Pré-condição | Passos | Resultado Esperado | Resultado Real | Status (PASS/FAIL)
  • Teste de falha importante: desconecte o sensor durante operação — o sistema trava ou se recupera?
  • Medir latência: long t0 = millis(); enviarDados(); long lat = millis() - t0; Serial.println("Latência: " + String(lat) + "ms");
  • GitHub: crie repositório público, faça commit com mensagem descritiva, adicione README com: objetivo, hardware necessário e como rodar
  • Backup da demo: grave um vídeo de 2 min do sistema funcionando — se der problema ao vivo, você tem prova que funcionou!
