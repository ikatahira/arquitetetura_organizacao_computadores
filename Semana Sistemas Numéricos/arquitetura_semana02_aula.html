BINARY: Sistemas Numéricos — Conversão de Bases

Computadores existem em base 2 — transistores que ligam ou desligam. Humanos pensam em base 10. Programadores leem memória em base 16. Sistemas embarcados e protocolos de rede usam base 8. Saber transitar entre essas bases fluidamente é a habilidade que separa quem usa computadores de quem entende computadores. Nesta missão você vai dominar cada conversão com simuladores passo a passo e exercícios práticos.

1. POR QUE MÚLTIPLAS BASES?
// antes de converter, entenda o motivo de cada base existir

// O que é uma base numérica?

Uma base define quantos símbolos diferentes um sistema usa e qual é o peso de cada posição. No decimal (base 10), os símbolos são 0–9 e cada posição vale 10× mais que a anterior. No binário (base 2), só existem 0 e 1, e cada posição vale 2× mais. A posição mais à direita sempre vale base⁰ = 1.

O número 347 em decimal significa: 3×10² + 4×10¹ + 7×10⁰ = 300 + 40 + 7 = 347. O mesmo princípio funciona para qualquer base — só mudam o símbolo e o peso.

// Por que computadores usam binário?

Transistores operam com dois estados elétricos: tensão alta (~5V ou 3.3V) = 1, tensão baixa (~0V) = 0. É impossível distinguir 10 níveis de tensão de forma confiável em bilhões de transistores. Dois níveis são robustos, rápidos e miniaturizáveis. Por isso tudo dentro de um processador é binário.

Curiosidade real: Um chip M3 da Apple tem 19 bilhões de transistores. Cada um pode estar em 0 ou 1. O resultado é toda a computação moderna — vídeos, jogos, IA — construída sobre bilhões de chavezinhas ligadas/desligadas.

// Por que octal (8) e hexadecimal (16)?

Binário é verboso para humanos. 11001010 é um byte — 8 dígitos para representar apenas o número 202. Hexadecimal resolve isso: cada dígito hex representa exatamente 4 bits, então 1 byte vira 2 dígitos: CA. Octal usa 3 bits por dígito e era popular em sistemas mais antigos (Unix, endereçamento de memória).

Regra de ouro: Hex e octal são atalhos para binário — não são sistemas independentes. Aprender a ir de binário para hex em um segundo é o objetivo real desta aula.

// DECIMAL → BINÁRIO: Método das Divisões Sucessivas por 2

Regra: Divida o número por 2 repetidamente. Anote o resto de cada divisão (sempre 0 ou 1). O resultado binário é lido de baixo para cima (do último resto ao primeiro).

O primeiro resto obtido é o bit menos significativo (LSB — Least Significant Bit), o último é o mais significativo (MSB — Most Significant Bit).

// Exemplo resolvido: 42₁₀ → Binário

1
42 ÷ 2 = 21
resto 0 ← LSB (bit 0)
2
21 ÷ 2 = 10
resto 1 ← bit 1
3
10 ÷ 2 = 5
resto 0 ← bit 2
4
5 ÷ 2 = 2
resto 1 ← bit 3
5
2 ÷ 2 = 1
resto 0 ← bit 4
6
1 ÷ 2 = 0
resto 1 ← MSB (bit 5)
42₁₀ = 101010₂ (leia os restos de baixo para cima: 1→0→1→0→1→0)
Verificação: 1×2⁵ + 0×2⁴ + 1×2³ + 0×2² + 1×2¹ + 0×2⁰ = 32 + 0 + 8 + 0 + 2 + 0 = 42 ✓

// BINÁRIO → DECIMAL: Método dos Pesos Posicionais

Regra: Multiplique cada bit pelo seu peso posicional (potência de 2), contando da direita para esquerda a partir de 2⁰. Some todos os produtos onde o bit é 1.

Bits com valor 0 não contribuem para a soma — você só precisa somar as potências onde aparecer o dígito 1.

10110101₂ = 181₁₀ — Apenas some as potências onde o bit é 1: 128 + 32 + 16 + 4 + 1 = 181.

// HEXADECIMAL → DECIMAL: Método dos Pesos Posicionais (base 16)

Regra: Multiplique cada dígito hex pelo seu peso (potência de 16). Some tudo. Lembre: A=10, B=11, C=12, D=13, E=14, F=15.

// Exemplo resolvido: 2F₁₆ → Decimal

Dígito2F (=15)
Peso (16ⁿ)16¹ = 1616⁰ = 1
Produto2 × 16 = 3215 × 1 = 15
32 + 15 = 47₁₀

// Exemplo resolvido: A3F₁₆ → Decimal

DígitoA (=10)3F (=15)
Peso (16ⁿ)16² = 25616¹ = 1616⁰ = 1
Produto10 × 256 = 25603 × 16 = 4815 × 1 = 15
2560 + 48 + 15 = 2623₁₀

// Exemplo resolvido: FF₁₆ → Decimal

DígitoF (=15)F (=15)
Peso16¹ = 1616⁰ = 1
Produto15 × 16 = 24015 × 1 = 15
240 + 15 = 255₁₀
FF em contexto: 0xFF = 255 = o valor máximo de um byte. Cores RGB usam 3 bytes, então (255, 255, 255) = #FFFFFF = branco puro. (0, 0, 0) = #000000 = preto.

// BINÁRIO ↔ HEXADECIMAL: O atalho mais poderoso

Como 16 = 2⁴, cada dígito hex corresponde exatamente a 4 bits (1 nibble). Não há cálculo intermédio — basta agrupar ou expandir.

BIN → HEX: Agrupe os bits em grupos de 4 da direita para a esquerda. Converta cada grupo usando a tabela.
HEX → BIN: Expanda cada dígito hex em 4 bits usando a tabela.

// OCTAL: Base 8 e sua relação com binário

Como 8 = 2³, cada dígito octal corresponde a exatamente 3 bits. A lógica é idêntica ao hexadecimal, mas com grupos de 3 em vez de 4.

BIN → OCT: Agrupe bits em grupos de 3 da direita para a esquerda. Converta cada grupo (0–7).
OCT → BIN: Expanda cada dígito em 3 bits.

// Exemplo: 45₁₀ → Octal (e de volta a binário)

O octal é um ótimo intermediário para verificar conversões binárias, pois cada dígito representa exatamente 3 bits sem ambiguidade.

resto 5 ← LSB
2
5 ÷ 8 = 0
resto 5 ← MSB
45₁₀ = 55₈ (0o55)

Agora expandindo cada dígito octal em 3 bits: 5 = 101, 5 = 101

55₈ = 101101₂ = 45₁₀ ✓ (verificação cruzada funcionou!)

LEITURA DE DADOS REAIS — interprete, não apenas converta
+90 XP⏱ 35min
Interprete dados hexadecimais de contextos reais. A ideia é conectar a teoria com situações que você vai encontrar como desenvolvedor ou administrador de sistemas.
  • 1.A cor CSS #FF6B35 — converta cada canal (R, G, B) para decimal. Qual cor predomina? Qual canal está no máximo?
  • 2.Um dump de memória mostra os bytes: 48 65 6C 6C 6F (em hex). Converta cada byte para decimal. Consulte a tabela ASCII e descubra a palavra armazenada.
  • 3.Um protocolo de rede transmite o valor octal 0o377. Converta para decimal e binário. Qual é a relação desse valor com o byte máximo?
  • 4.Um endereço IPv4 tem a máscara de sub-rede 255.255.255.0. Escreva cada octeto em binário. Por que esta máscara define 256 endereços possíveis?
ASCII: A=65=0x41, B=66=0x42 ... Z=90=0x5A. Letras minúsculas: a=97=0x61 ... z=122=0x7A. Espaço=32=0x20. Números: 0=48=0x30 ... 9=57=0x39.

DESAFIO — conversões encadeadas e raciocínio inverso
+80 XP⏱ 25min
Questões que testam se você realmente entendeu os métodos, não apenas memorizou. Cada uma requer pensar antes de calcular.
  • 1.Qual é o menor número de bits necessários para representar o decimal 1000? Justifique usando potências de 2.
  • 2.Um número binário de 8 bits tem os 4 bits mais significativos iguais a 1010 e os 4 menos significativos iguais a 0011. Qual é seu valor em decimal e hexadecimal?
  • 3.Em binário, 10000000 representa 128. Quanto representa 01111111? E 11111111? Existe um padrão? Explique.
  • 4.Converta os 3 bytes do endereço MAC parcial A4:C3:F0 para binário. Quantos bits tem um endereço MAC completo de 6 bytes?
Para o item 1: log₂(1000) ≈ 9.96, portanto são necessários 10 bits (2¹⁰ = 1024 ≥ 1000). Para o item 3: pense nos pesos e no que acontece quando você some 1 ao máximo e ao mínimo.
