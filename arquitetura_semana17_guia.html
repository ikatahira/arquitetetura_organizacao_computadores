01 · Cliente HTTP
✏️ Exercício 1 — Cliente HTTP com ESP32
📋 Objetivo: Buscar e interpretar dados de uma API externa via WiFi
⏱ 30 min
CritérioInsuficiente (0–5)Básico (6–7)Proficiente (8–9)Excelente (10)
Conexão WiFiNão conecta à redeConecta mas sem verificação de statusConecta com loop de espera e exibe IP localReconecta automaticamente em caso de queda com millis()
Requisição HTTP GETNão faz requisiçãoFaz mas não trata código de respostaVerifica HTTP 200 e lê payload completoTrata redirecionamentos e erros (404, 500) com mensagens claras
Parsing JSONNão parseia JSONAcessa campo fixo sem verificar erroUsa ArduinoJson com documento dimensionado corretamenteValida campos ausentes e exibe todos os campos relevantes
Exibição dos DadosSem saída legívelImprime JSON bruto no serialFormata saída com rótulos e unidades (°C, km/h, etc.)Exibe em display OLED ou cria log com timestamp
Conexão com TeoriaSem conexão teóricaMenciona que usa InternetRelaciona ao modelo cliente-servidor e pilha TCP/IPExplica como o DMA do WiFi libera a CPU durante a recepção de dados
💡 ORIENTAÇÕES E DICAS
  • Conectar: WiFi.begin("SSID", "senha"); enquanto WiFi.status() != WL_CONNECTED aguarde com delay(500)
  • API gratuita de clima: http://wttr.in/Presidente+Prudente?format=j1 — retorna JSON com temperatura
  • HTTPClient: http.begin(url); int code = http.GET(); String payload = http.getString();
  • ArduinoJson: DynamicJsonDocument doc(4096); deserializeJson(doc, payload);
  • Conexão teórica: a requisição HTTP é I/O por interrupção — o WiFi avisa quando chegam bytes, a CPU não fica em polling!
#include <WiFi.h>
#include <HTTPClient.h>
#include <ArduinoJson.h>
WiFi.begin(ssid, pass);
while (WiFi.status() != WL_CONNECTED) delay(500);
HTTPClient http;
http.begin("http://wttr.in/?format=j1");
if (http.GET() == 200) {
  DynamicJsonDocument doc(4096);
  deserializeJson(doc, http.getString());
  Serial.println(doc["current_condition"][0]["temp_C"].as<String>());
}
02 · Servidor Web Embarcado
✏️ Exercício 2 — Servidor Web no ESP32
📋 Objetivo: Inverter os papéis — ESP32 como servidor HTTP
⏱ 35 min
CritérioInsuficiente (0–5)Básico (6–7)Proficiente (8–9)Excelente (10)
Inicialização do ServidorServidor não iniciaInicia mas sem rotas definidasServidor na porta 80 com rota raiz funcionalExibe IP no serial e em display; trata rota 404
Rotas de ControleSem rotas /on e /offUma rota funciona, outra nãoRotas /on e /off controlam LED e retornam HTMLAdiciona rota /toggle e resposta com estado atual em JSON
Rota /status (JSON)Sem rota de statusRetorna texto sem formatoRetorna JSON com led, uptime e IP corretosInclui RSSI WiFi, memória livre e versão do firmware
Interface HTMLSem HTMLHTML mínimo sem estiloPágina com botões funcionais e feedback visualInterface responsiva com CSS inline e atualização automática via JS
Qualidade do CódigoCódigo confuso sem comentáriosFunciona mas sem organizaçãoCódigo organizado com funções separadas por rotaUsa lambdas no on() e separa HTML em String constante
💡 ORIENTAÇÕES E DICAS
  • Biblioteca: #include <WebServer.h> — crie WebServer server(80);
  • Registre rotas: server.on("/on", []() { digitalWrite(2, HIGH); server.send(200, "text/plain", "LED ON"); });
  • No loop(): server.handleClient(); — isso é o "despachante" de requisições HTTP
  • JSON de status: server.send(200, "application/json", "{\"led\":" + String(digitalRead(2)) + ",\"uptime\":" + String(millis()/1000) + "}");
  • Atenção: WebServer.h é síncrono — uma requisição por vez. Para múltiplas conexões simultâneas, use ESPAsyncWebServer
#include <WebServer.h>
WebServer server(80);
server.on("/", []() {
  server.send(200, "text/html",
  "<h1>ESP32</h1><a href='/on'>ON</a> <a href='/off'>OFF</a>");
});
server.begin();
// no loop(): server.handleClient();
03 · Dashboard IoT
✏️ Exercício 3 — Envio de Dados para Nuvem
📋 Objetivo: Publicar dados de sensores em plataforma IoT
⏱ 25 min
CritérioInsuficiente (0–5)Básico (6–7)Proficiente (8–9)Excelente (10)
Leitura do SensorNão lê sensor algumLê mas valor não faz sentidoLê ADC e converte para grandeza física real (°C ou %)Aplica média de 10 amostras para reduzir ruído do ADC
Envio PeriódicoNão envia dadosEnvia com delay() bloqueanteEnvia a cada 15s com millis() sem bloquear loopImplementa fila de envio para não perder leituras durante falha
Integração com PlataformaSem integraçãoIntegra mas dados não aparecem no dashboardDados visíveis no gráfico da plataforma em tempo realConfigura alertas ou triggers na plataforma (ex: temperatura alta)
Medição de LatênciaSem mediçãoMede tempo mas não reportaExibe latência de cada envio no serial com millis()Calcula média e desvio padrão da latência ao longo de 10 envios
DocumentaçãoSem documentaçãoApenas screenshot do gráficoLog serial completo + URL do dashboard compartilhávelRelatório comparando HTTP POST vs MQTT com dados de latência
💡 ORIENTAÇÕES E DICAS
  • ThingSpeak: crie canal gratuito em thingspeak.com, anote a API Key de escrita
  • Envio via HTTP POST: http.begin("http://api.thingspeak.com/update?api_key=KEY&field1=" + String(valor));
  • MQTT com HiveMQ: #include <PubSubClient.h> — broker gratuito em broker.hivemq.com:1883
  • Temperatura interna do ESP32: float t = temperatureRead(); — sem sensor externo!
  • Conexão com teoria: MQTT usa pub/sub, um padrão de I/O assíncrono — a CPU não fica esperando confirmação!
